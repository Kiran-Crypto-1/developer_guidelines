Engineering Philosophy

Fundamentals over frameworks, clarity over cleverness, ownership over delegation

Engineering at EchorTech prioritizes fundamentals over frameworks, clarity over cleverness, and ownership over delegation. This philosophy ensures that every engineer can read, reason about, debug, test, and evolve existing systems confidently.

Core Principle

Strong fundamentals mean the ability to read, reason about, debug, test, and evolve existing systems confidently. Every engineer is accountable for the quality, stability, and long-term health of the code they contribute.

The Context Check: Mandatory Pre-Development Ritual

Before writing a single line of code, every developer must complete this context check. This transforms "ticket closers" into "domain-aware engineers" who understand the business impact of their work.

1. Identify The Domain

What industry or business domain does this feature serve?

  • Fintech: Financial transactions, payments, trading systems
  • Healthcare: Patient data, medical records, HIPAA compliance
  • Logistics: Shipping, inventory, real-time tracking
  • E-commerce: Product catalogs, checkout, order management
2. Define The Currency of Trust

What is the most critical quality attribute for this domain?

  • Fintech: Precision (decimal math accuracy), Audit Trails (immutable transaction logs)
  • Healthcare: Privacy (PII/PHI protection), Access Control (role-based permissions)
  • Logistics: Offline Capability (works without internet), Sync Conflict Resolution (handles concurrent updates)
  • Trading Systems: Speed (latency matters), Idempotency (duplicate requests are safe)
3. Understand The Stakes

What happens if this feature breaks in production?

  • Do we lose money? (Payment processing failure)
  • Do we violate regulations? (HIPAA breach, PCI-DSS violation)
  • Do we lose customer trust? (Data leak, privacy violation)
  • Do we just annoy users? (UI glitch, slow loading)

The stakes determine your testing strategy, monitoring approach, and deployment process.

Remember

A developer who doesn't understand the domain-specific risks (e.g., "Privacy" in Health vs. "Speed" in Trading) will inevitably build fragile or dangerous software. The Context Check prevents this.

Responsibility & Ownership

End-to-end accountability from design to deployment

Definition of Ready: The Ownership Standard

Before beginning development, you must be able to answer these questions:

  • Who uses this feature? Identify the end-user persona, their technical proficiency, and their primary use case.
  • What breaks if this fails? Understand the failure modes: Does it block critical workflows? Does it expose sensitive data? Does it cause financial loss?
  • How will I verify it works in production? Define your verification strategy: monitoring, logging, user testing, or automated checks.

End-to-End Ownership

Developers own their changes from conception to production monitoring:

Design Phase

  • Clarify requirements and edge cases before coding
  • Identify ambiguity and seek clarification
  • Consider failure modes and error handling

Implementation Phase

  • Write code that you can explain to a junior developer
  • Document complex logic and business rules
  • Consider performance and scalability

Testing Phase

  • Write tests before or alongside code
  • Test failure modes, not just happy paths
  • Verify edge cases and boundary conditions

Deployment & Monitoring

  • Deploy and verify in production yourself
  • Monitor metrics and logs post-deployment
  • Be available for immediate rollback if needed

Communication & Explanation

Every engineer must be able to explain:

  • What their code does Clear, concise explanation of functionality and purpose.
  • Why it exists The business problem it solves and the user need it addresses.
  • How it can fail Known failure modes, edge cases, and potential bugs.
  • How it can be debugged Logging strategy, error messages, and troubleshooting steps.

Incident Culture

If a defect escapes to production: The focus is on understanding why it happened and preventing recurrence, not assigning blame. We fix the system, not the person. Every incident is a learning opportunity.

No "Throwing Over the Wall"

You cannot write code, push it, and assume someone else will handle deployment, testing, or production issues. Ownership means you are responsible for the entire lifecycle of your changes.

Testing & Quality Standards

Testing is mandatory, not optional

Mandatory Responsibility

Testing is a mandatory developer responsibility. It is not delegated to QA, not optional, and not something you can skip "because it's a small change."

Testing Performance Targets

Unit Tests: Execute in < 100ms per test | Integration Tests: Execute in < 5s per test | E2E Tests: Execute in < 30s per test. Test suites should complete in < 5 minutes total.

The Testing Pyramid

Apply appropriate testing based on context and risk. Use this decision framework:

Testing Decision Framework

Write Unit Tests When:
  • Testing business logic in isolation
  • Validating data transformations
  • Checking edge cases and boundary conditions
  • Verifying algorithm correctness
Write Integration Tests When:
  • Testing API endpoints with database
  • Verifying service-to-service communication
  • Validating data persistence
  • Testing error handling and rollback scenarios
Write E2E Tests When:
  • Testing critical user workflows (login, checkout, payment)
  • Validating complete business processes
  • Testing cross-browser compatibility
  • Verifying user experience end-to-end

Unit Tests

When: Test individual functions, methods, or components in isolation.

  • Mock external dependencies (APIs, databases, file systems)
  • Test business logic and data transformations
  • Fast execution (milliseconds per test)
  • High coverage of edge cases and error paths

Tools: Jest, Vitest, pytest, Go testing package

Integration Tests

When: Test interactions between components (API + Database, Service + Queue).

  • Use Docker containers for databases and external services
  • Test real API endpoints with test data
  • Verify data persistence and retrieval
  • Test error handling and rollback scenarios

Tools: Supertest, pytest with fixtures, Docker Compose

End-to-End (E2E) Tests

When: Test critical user flows from start to finish.

  • Test complete workflows (e.g., User Registration â†’ Login â†’ Purchase â†’ Payment)
  • Verify UI interactions and user experience
  • Test in production-like environments
  • Focus on high-value, high-risk paths

Tools: Cypress, Playwright, Selenium

Common Testing Mistakes to Avoid

  • Testing Implementation Details: Don't test that a function calls another function. Test the behavior and outcome.
  • Over-Mocking: Mock external dependencies, but test real logic. Too many mocks hide bugs.
  • Testing Framework Features: Don't test React, test your components. Don't test Express, test your routes.
  • Ignoring Flaky Tests: Fix or remove flaky tests. They erode trust in the test suite.
  • Testing Only Happy Paths: Most bugs occur in error conditions. Test failures, edge cases, and invalid inputs.

Testing Failure Modes: The "Negative" Path

Don't just test the happy path. Every feature must be tested against failure scenarios. Related: This aligns with your Context Check (Section 1) - high-stakes features need more comprehensive failure testing.

  • Network Failures What happens when the API times out? When the database connection drops? When Redis is unavailable?
  • Invalid Input Test with null, undefined, empty strings, special characters, SQL injection attempts, XSS payloads.
  • Edge Cases Empty lists, zero values, negative numbers, extremely large numbers, unicode characters, timezone boundaries.
  • Concurrency Issues Race conditions, duplicate requests, simultaneous updates, deadlocks.
  • Resource Exhaustion Memory limits, disk space, rate limits, connection pool exhaustion.

Code Completeness Criteria

A feature is considered incomplete if:

  • Its failure modes have not been evaluated and tested
  • Edge cases are not handled or documented
  • It passes code review but fails basic scenarios in testing
  • Error messages are unclear or unhelpful
  • There is no way to verify it works in production

Manual Verification

For some scenarios (UI polish, complex user workflows, visual regression), documented manual verification is acceptable. However, this must be explicitly documented with step-by-step instructions and expected outcomes.

Responsible LLM Usage

LLMs are productivity tools, not sources of truth

Critical Rule

All LLM-generated outputs must be validated by execution, testing, or reasoning through concrete examples before use. Never copy-paste AI-generated code directly into production without review.

The "Context Sandwich" Prompt Pattern

Instead of asking vague questions like "How do I do X?", structure your prompts with clear context:

// âŒ BAD: Vague, no context "How do I create an S3 bucket?" // âœ… GOOD: Context Sandwich Pattern 1. Role: "Act as a Senior AWS Architect with expertise in HIPAA compliance..." 2. Context: "We are building a healthcare application that stores patient medical records. We need to ensure encryption at rest, audit logging, and strict access controls..." 3. Task: "Write a Terraform script to create an S3 bucket with the following requirements..." 4. Constraints: "Ensure encryption is enabled (SSE-S3 or KMS), public access is blocked, versioning is enabled, and lifecycle policies are configured for compliance retention."

Best Practices

  • Use clear task descriptions with specific requirements
  • Break large tasks into smaller, focused chunks
  • Provide only relevant context (don't paste entire codebases)
  • Use step-by-step prompting for complex problems
  • Request constrained output formats (JSON, specific code patterns)

What to Avoid

  • Vague or ambiguous prompts
  • Including secrets, credentials, or API keys
  • Pasting entire files without context
  • Asking for code without specifying constraints
  • Trusting AI output without validation

The "Adversary" Prompt: Self-Audit Before Code Review

Before pushing code, use this prompt pattern to self-audit for security and quality issues:

// Example: Security Audit "I just wrote this function to handle user authentication. Act as a security adversary and tell me: 1. How can this be exploited via SQL injection? 2. What happens if an attacker sends malformed input? 3. Are there any race conditions or timing attacks? 4. What sensitive data might be leaked in error messages?" // Example: Code Quality Audit "Review this code for: - Performance bottlenecks - Memory leaks - Error handling gaps - Edge cases that aren't handled"

Data Hygiene: Never Include Sensitive Data

Never Include in Prompts

  • Secrets: API keys, passwords, tokens, private keys
  • Credentials: Database connection strings, AWS access keys, OAuth secrets
  • Client PII: Names, emails, phone numbers, addresses
  • PHI (Protected Health Information): Medical records, patient data, health identifiers
  • Financial Data: Credit card numbers, bank account details, transaction IDs

If you need to test with sensitive data patterns, use anonymized examples or synthetic data that mimics the structure without exposing real information.

Validation Strategy

For any AI-generated code, especially for critical operations, follow this validation process:

  • Generate Tests First Ask the AI to generate unit tests (including failure cases) before using the code.
  • Review Logic Read through the code and understand what it does. Don't trust it blindly.
  • Test in Isolation Run the code in a test environment with various inputs, including edge cases.
  • Verify Against Requirements Ensure the code actually solves the problem you asked it to solve.
  • Check for Security Issues Use the "Adversary" prompt or security scanning tools.

React & React Native Fundamentals

Deep understanding of React's rendering and state management:

Component Lifecycle & Rendering

  • Component Lifecycle: Mount, update, unmount phases (class components and hooks equivalents)
  • Re-renders: When and why components re-render, how to identify unnecessary re-renders
  • Memoization: useMemo for expensive calculations, useCallback for function references
  • Keys: Why keys matter in lists, how they affect reconciliation

State Management

  • useState: When to use, batching updates, functional updates
  • useEffect: Dependencies array, cleanup functions, avoiding infinite loops
  • Context: When to use, performance implications, avoiding prop drilling
  • Complex State: useReducer, state machines, external state libraries (Zustand, Redux)

Critical Understanding

You must be able to explain why a component re-renders and how memoization (useCallback, useMemo) affects performance. If you can't explain it, you shouldn't use it.

Common React Mistakes

  • Overusing useMemo/useCallback: These have overhead. Only use when profiling shows a performance issue.
  • Creating functions in render: Pass stable function references to child components.
  • Missing keys in lists: Always provide stable, unique keys. Index as key is an anti-pattern.
  • State in wrong place: Lift state up only when needed. Don't over-lift.
  • useEffect without dependencies: Missing dependencies cause stale closures. Include all dependencies.
  • Fetching in useEffect: Use React Query or SWR for data fetching, not useEffect.

Troubleshooting: Why is my component re-rendering?

Check these in order:
  1. Parent re-rendered: Use React DevTools Profiler to see render tree
  2. State changed: Check if state/props actually changed (use Object.is or deep comparison)
  3. Context value changed: Context providers re-render all consumers when value changes
  4. Function reference changed: Inline functions create new references on every render
  5. Object/Array reference changed: New object/array literals create new references

Solution: Use React.memo, useMemo, useCallback, or move state down to reduce re-render scope.

Backend, Data & Systems Engineering

Database design, async processing, and failure-aware architecture

SQL Fundamentals

Strong SQL knowledge is required for relational database work:

  • Table Design Normalization (1NF, 2NF, 3NF), denormalization trade-offs, primary/foreign keys, constraints.
  • Indexing When to index, composite indexes, covering indexes, index maintenance overhead.
  • Query Optimization EXPLAIN plans, identifying slow queries, avoiding N+1 problems, using JOINs efficiently.
  • Transactions ACID properties, isolation levels, deadlocks, rollback strategies.

Database Performance Targets

Read Queries: < 50ms p95 | Write Queries: < 200ms p95 | Connection Pool: Monitor utilization, keep < 80% | Query Timeout: Set appropriate timeouts (typically 5-30s)

SQL vs NoSQL: Understanding Trade-offs

Decision Framework: Use this criteria to choose between SQL and NoSQL databases.

SQL vs NoSQL Decision Tree

Choose SQL (PostgreSQL, MySQL) when:
  • You need complex queries with JOINs and aggregations
  • ACID transactions are critical (financial transactions, payments)
  • Data relationships are important (normalized data model)
  • Data consistency is paramount (can't tolerate eventual consistency)
  • You need strong schema enforcement
  • Reporting and analytics are primary use cases
Choose NoSQL (MongoDB, DynamoDB) when:
  • You need horizontal scaling (millions of requests per second)
  • Schema is flexible or evolves frequently
  • Simple key-value or document access patterns
  • High write throughput is required
  • Eventual consistency is acceptable
  • You need geographic distribution with low latency

Remember: Many systems use both. Use SQL for transactional data, NoSQL for high-scale read/write operations.

SQL (PostgreSQL, MySQL)

Use when:

  • Complex queries with joins and aggregations
  • ACID transactions are critical
  • Structured, relational data

NoSQL (MongoDB, DynamoDB)

Asynchronous Processing

Understanding event-driven systems and message queues:

  • Event-Driven Systems Event sourcing, CQRS, event ordering, idempotency, event replay.
  • AWS SQS Standard vs FIFO queues, visibility timeout, dead-letter queues, long polling.
  • Message Patterns Pub/sub, request/reply, fan-out, message routing, error handling.

Async & Concurrency Models

Node.js

  • Event loop, non-blocking I/O
  • Worker threads for CPU-intensive tasks
  • Async/await, Promises, callbacks

  • GIL limitations, asyncio for I/O
  • multiprocessing for CPU-bound tasks

Design Systems with Failure in Mind

Every system will fail. Design for graceful degradation and recovery:

  • Retries Exponential backoff, jitter, maximum retry limits, circuit breakers.
  • Idempotency Design operations to be safely retryable, use idempotency keys for critical operations.
  • Partial Failures Handle cases where some operations succeed and others fail, implement compensation logic.
  • Rollback Strategies Database transactions, saga pattern, compensating transactions, feature flags for gradual rollback.

Communication, Security & Professional Conduct

Team communication, security practices, and professional standards

Communication Standards

Mandatory Channel Monitoring

Reading and acknowledging messages in mandatory channels (scrum, general) is required. Missed communication is not an acceptable reason for delivery or coordination failures.

  • Active Participation Respond to direct mentions within reasonable timeframes, acknowledge important announcements, participate in team discussions.
  • Status Updates Provide regular updates on work progress, blockers, and completion status in designated channels.
  • Documentation Document decisions, architecture changes, and important discussions. Don't rely solely on chat history.
  • Escalation Know when to escalate issues. Don't struggle in silence when blocked or facing critical problems.

Security & Data Protection

  • Secure Backup Practices Work must be backed up securely, including environment configurations and secrets. Use version control for all code, never store secrets in code repositories.
  • Secret Management Use secure secret management tools (AWS Secrets Manager, HashiCorp Vault, environment variables). Never commit secrets, API keys, or credentials to version control.
  • Access Control Follow principle of least privilege. Only request access to resources you need. Regularly review and revoke unnecessary access.
  • Data Handling Understand data classification (public, internal, confidential, restricted). Handle sensitive data according to company policies and regulatory requirements (HIPAA, PCI-DSS, GDPR).
  • Security Updates Keep dependencies updated, patch known vulnerabilities promptly, use security scanning tools in CI/CD pipelines.

Professional Conduct

Company Resources

Company laptops, licensed software, and paid tools are strictly for professional use only. Personal projects, unauthorized software, or misuse of company resources is prohibited.

  • Code of Conduct Treat all team members with respect, provide constructive feedback, maintain professional communication in all channels.
  • Time Management Be punctual for meetings, respect deadlines, communicate proactively if timelines need adjustment.
  • Continuous Learning Stay updated with industry best practices, share knowledge with the team, contribute to documentation and knowledge base.
  • Accountability Take ownership of mistakes, learn from failures, contribute to post-mortems and process improvements.
